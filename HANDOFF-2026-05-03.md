## Where we landed (architecture as of iter 9)

```
Director (selectors only) → Playwright (record-time live bbox) → Remotion (paint + zoom)
```

- Director writes `target_selector` per beat. **No static bbox** in scene.json.
- `record/runner.ts` calls `page.locator(target_selector).boundingBox()` at each beat's `t_ms` and writes `<hash>.bboxes.json` alongside the webm.
- `commands/compose.ts` loads bboxes.json + splices live bbox into BOTH `action.highlight.bbox` AND `action.zoom.bbox` (when `zoom.scale !== 1.0`). Same coord frame as the recording → zero drift.
- `Spotlight.tsx` + `HighlightFrame.tsx` paint at given bbox. Pure dumb painter.

## Open problems (carry forward)

### Problem 1 — Two-level script/visual alignment

**Layer A: scripts are not role-aware.**

WorkBlock has 4+ roles: `super_admin`, `coordinator`, `team_lead`, `team_member`. Each role sees a *different* dashboard. A coordinator sees "Coordinator Queue 9", a team member sees their personal backlog only. Today the script writer agent gets ONE generic script per page.

**Required behavior:** before script generation, the director runs a *discovery phase*:
1. Ingest codebase markdown — `CLAUDE.md`, `Workblock/40-Codebase/*.md`, `docs/*.md`. The plugin already has access to local files; CLAUDE.md should drive what to read.
2. Ask the user for an additional context folder if not auto-detected. Or pull from claude-mem.
3. Inspect each route under multiple authenticated sessions (per role) and capture per-role DOM/inspect data.
4. Generate **one script + scene per (page × role)**.
5. Optionally generate a "common" script for content shared across roles (e.g., page header).

This is a new **Phase 0 — discovery** stage upstream of `scan` / `plan` / `script-prepare`.

**Layer B: scene zoom/highlight don't match what the narration says.**

Observed in iter 9 dashboard segment: narration says "go to the backlog and pick up a task" while the camera zooms into the Pick Up button on the dashboard's Suggested Tasks card. Two different actions. The script writer wrote one story; the scene director chose a different focal element. This will compound when audio TTS ships — voice will diverge from visuals.

**Required fix:**
- Tighter coupling between writer + director. Either:
  - Single agent produces narration + scene together (writer + director merged), OR
  - Writer's `alignments[]` array becomes a CONTRACT — director MUST anchor each beat on the element the writer named in the corresponding phrase, OR the consume step rejects.
- Add a verification pass: after scene generation, an "alignment auditor" subagent checks each beat's narration phrase mentions the focal element by name; flags mismatches.

### Problem 2 — TTS

Stick with Gemini 2.5 Flash Preview TTS for now. Free-tier daily limit (10 req/day) burned through twice today. Issues:
- API returns raw PCM L16 (not MP3) — already wrapped to WAV in `tts/gemini.ts`. Works.
- SSML breaks/emphasis tags cause early truncation — already switched to plaintext prompt. Long inputs still truncated by the preview model (s01 75s narration → 36s output).
- Daily quota: 10 requests, ~5 segments per typical run → won't get through one full pass without paid tier upgrade.

**Action:** consider paid Gemini tier OR test alternative provider (`SAR AI` / ElevenLabs / Azure Neural) as fallback.
